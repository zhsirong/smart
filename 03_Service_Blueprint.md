# Service Blueprint（业务流程与服务蓝图）

## 1. 文档目标

将 SRP 中的 As-Is / To-Be 业务动作转成更适合产品、设计、研发、业务共同对齐的服务蓝图。

## 2. As-Is（现状流程）

### 2.1 客户侧动作
1. 客户发送邮件
2. 邮件中附带 invoice / bank statement / receipt 等文件

### 2.2 会计侧动作
1. 打开邮件
2. 识别附件类型
3. 下载附件
4. 上传到 file.ai
5. 等待 OCR 结果
6. 人工核对 OCR
7. 复制结果到 Xero
8. 月结时发现缺票
9. 翻查历史邮件找附件
10. 手工写催票邮件
11. 跟踪是否回复
12. 手工检查月结 checklist

### 2.3 现状痛点
- 多次切系统
- 信息不连续
- 低价值重复工作占比高
- 依赖老员工经验
- 缺少透明状态

## 3. To-Be（目标流程）

### 3.1 前台用户动作
1. 在工作台查看新邮件处理结果
2. 只对低置信度记录做人工复核
3. 查看凭证覆盖率和缺票列表
4. 一键触发/确认催票邮件
5. 查看月结 checklist 与预警

### 3.2 后台 Agent 动作
1. 轮询/订阅收件箱
2. 识别新邮件与附件
3. 调用 file.ai 做 OCR
4. 接收结果并判断置信度
5. 调用 Xero 写入凭证
6. 更新凭证状态表
7. 覆盖率不足时检索历史邮件
8. 调用邮件服务生成/发送催票邮件
9. 更新催票记录
10. 检查月结 checklist

## 4. 服务蓝图分层

| 层级 | 内容 |
|---|---|
| 用户可见层 | 新邮件处理结果、异常待复核、覆盖率、缺票、催票状态、checklist 状态 |
| 前台交互层 | 查看详情、确认字段、编辑字段、生成催票、发送/审批、查看审计 |
| 后台编排层 | 邮件采集、OCR 调用、Xero 回写、检索、规则引擎、状态引擎 |
| 支撑系统层 | 邮箱、file.ai、Xero、规则库、审计日志 |

## 5. Blueprint 表

| 阶段 | 客户动作 | 会计动作 | 系统动作 | 审核门 | 输出 |
|---|---|---|---|---|---|
| 收件 | 发送邮件 | 无 | 读取邮件与附件 | 无 | 新邮件任务 |
| 识别 | 无 | 无 | 分类附件 | 无 | 附件类型 |
| OCR | 无 | 无 | 触发 OCR | 无 | 结构化字段 |
| 回写 | 无 | 必要时复核 | 写入 Xero | 低置信度需人工 | 凭证记录 |
| 状态追踪 | 无 | 查看覆盖率 | 更新状态表 | 无 | 覆盖率看板 |
| 缺票检索 | 无 | 查看缺失项 | 检索历史邮件 | 无 / 待验证 | 候选发票 |
| 催票 | 回复邮件 | 审核或直接发送 | 生成催票邮件 | 默认建议保留审核门 [待验证] | 催票记录 |
| 清单检查 | 无 | 查看预警 | 执行 checklist 规则 | 某些异常需高级会计师 | 清单状态 |

## 6. 关键体验原则

1. **让系统接管重复步骤，让人只处理异常**
2. **所有自动动作都必须可追踪**
3. **所有高风险动作都必须可人工接管**
4. **所有关键状态必须在单一控制台可见**
5. **检索和催票必须围绕“当前缺什么”展开，而不是让用户自己找**

## 7. 关键异常路径

### 7.1 OCR 失败
- 系统标记解析失败
- 进入人工处理队列
- 不自动写入 Xero

### 7.2 Xero 写入失败
- 记录失败原因
- 进入重试或人工提交队列
- 不影响原始邮件记录

### 7.3 历史邮件未命中
- 标记未命中
- 生成人工检查建议
- 不自动发送高确定性措辞的催票邮件

### 7.4 催票邮件发送失败
- 记录发送失败状态
- 支持重发
- 通知会计专员
